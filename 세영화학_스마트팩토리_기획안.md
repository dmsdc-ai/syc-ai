## 1. 왜 스마트팩토리인가?

### 현재 상황 (추정)
- 생산 스케줄: 엑셀 or 경험 기반
- 배송 계획: 기사 경험 의존
- 데이터: 수기 기록 or 분산
- 설비 상태: 실시간 파악 어려움

### 스마트팩토리로 얻을 수 있는 것

| 분야 | 현재 | 목표 |
|------|------|------|
| 생산 계획 | 수작업, 경험 의존 | 알고리즘 기반 자동 스케줄링 |
| 배송 | 기사 감각 | 비용 최적화된 경로/배차 |
| 품질 | 사후 대응 | 불량 원인 추적, 예방 |
| 에너지 | 관리 안됨 | kWh/ton 모니터링, 최적화 |

### 정부 지원 활용 가능
- 스마트공장 지원사업: https://www.smart-factory.kr/
- 스마트제조혁신 지원사업: https://www.bizinfo.go.kr/
- 구축 비용 상당 부분 지원받을 수 있음

## 2. 우리 팀의 강점

우리 3형제는 스마트팩토리 구축에 필요한 역량이 다 갖춰져 있어.

```
     덕영 (시스템)  ←────→  준영 (현장)  ←────→  기영 (최적화)

     임베디드/PLC          10년+ 경험         알고리즘/데이터
     시스템 아키텍처        모든 업무 파악      DP/PMP 최적화
     배터리/전장            Ground Truth       빅데이터 분석
```

### 왜 우리가 직접 해야 하나?
1. **우리 상황을 가장 잘 아는 건 우리**
2. **벤더 종속 위험 없음** - 나중에 유지보수 비용 폭탄 없음
3. **노하우 축적** - 장기적으로 우리 자산이 됨
4. **비용 절감** - 오픈소스 활용, 정부 지원 활용

## 3. 역할 분담 제안

| 이름 | 역할 | 구체적 업무 | 시간 투입 |
|------|------|-----------|----------|
| **덕영 (나)** | 시스템/임베디드 자문 | PLC 데이터 수집 구조 설계, 아키텍처 리뷰, 벤더 기술 검토 | 주말 (월 2~4회) |
| **준영** | 현장 CEO / 도메인 오너 | 공정 제약 제공, 데이터 검증, 현실 적용 승인, 현장 피드백 | 상시 (현업 병행) |
| **기영** | 총괄 아키텍트 / 개발 | 문제 정의, 최적화 알고리즘 개발, 데이터 분석, 시스템 구축 | 상시 |

### 내 역할 상세 (덕영)

나는 다른 직장 다니고 있어서 주말에만 참여할 수 있어.
그래서 **"코딩하는 사람"이 아니라 "기술 자문/리뷰어"** 역할을 맡을게.

**내가 할 수 있는 것:**
- PLC/설비에서 데이터 어떻게 뽑을지 구조 잡기
- 전체 시스템 아키텍처 검토
- 실시간성, 안정성, 확장성 관점에서 리뷰
- 외부 벤더가 제안하는 기술이 적합한지 판단

## 4. 해결할 과제 목록

### 우선순위 제안

| 순위 | 과제 | 설명 | 예상 효과 |
|:----:|------|------|----------|
| 🥇 | **생산 일정 최적화** | 납기, 셋업 시간, 폭 활용률 고려한 자동 스케줄링 | 납기 준수율 ↑, 생산 효율 ↑ |
| 🥈 | **배송 스케줄링** | 인건비+연료비+감가상각 고려한 최적 경로/배차 | 배송비용 15~25% 절감 |
| 🥉 | **데이터 수집 기반** | 설비/생산/품질 데이터 통합 수집 | 모든 분석의 기반 |
| 4 | 불량 원인 추적 | 조건 조합 vs 불량률 상관관계 분석 | 불량률 감소 |
| 5 | 작업자 의존도 감소 | 과거 성공 조건 자동 추천 | 품질 균일화 |
| 6 | 에너지 최적화 | kWh/ton 분석 및 최적 운전점 도출 | 전기료 절감 |

### 🥇 생산 일정 최적화 상세

**목적함수 (최적화 목표):**
- 납기 지연 최소화
- 셋업/교체 횟수 최소화
- 폭 활용률 최대화
- 불량률 간접 최소화

**제약조건:**
- 기계별 생산 가능 폭
- 1사이클 2롤 구조
- 혼합생산 우선 정책
- 기계별 조합 품목 수 제한

→ **기영이의 DP/PMP 최적화 연구 경험이 그대로 적용되는 영역**

### 🥈 배송 스케줄링 상세

**목적함수:**
- 총 비용 최소화 (인건비 + 연료비 + 차량 감가상각)

**입력 데이터:**
- 당일 출하 목록 (중량, 부피, 파렛트 수)
- 배송 구역 / 거리
- 차량별 적재 한계

**결과:**
- 필요 차량 대수
- 각 차량 경로
- 출발 시간

→ **전형적인 OR(운영최적화) 문제, AI 없이도 해결 가능**

## 5. 활용할 기술 스택

### 오픈소스 도구 (무료)

| 분야 | 도구 | 설명 | 점수 |
|------|------|------|:----:|
| 생산 계획 | **frePPLe** | 오픈소스 APS, TOC 기반 스케줄링 | 8.55/10 |
| 배송 최적화 | **PyVRP** | State-of-the-art VRP 솔버 | 9.00/10 |
| 데이터 수집 | **ThingsBoard** | IoT 플랫폼, 다양한 프로토콜 지원 | 8.60/10 |
| PLC 통신 | **opcua-asyncio** | Python OPC-UA 라이브러리 | 8.15/10 |
| 모니터링 | **Libre** | Grafana 기반 제조 모니터링 | 8.45/10 |
| DB | **InfluxDB + PostgreSQL** | 시계열 + 마스터 데이터 | - |

### 시스템 아키텍처 (초안)

```
현장 (Shop Floor)
  PLC/센서 → OPC-UA (opcua-asyncio) → MQTT → ThingsBoard
  또는
  PLC → Raspberry Pi GPIO → InfluxDB
           ↓
데이터 레이어
  InfluxDB (시계열) + PostgreSQL (마스터 데이터)
           ↓
애플리케이션 레이어
  생산계획: frePPLe
  모니터링: Libre (Grafana 기반)
  배송최적화: PyVRP + OR-Tools
  분석: Python (Pandas, Scikit-learn)
           ↓
시각화/UI
  Grafana 대시보드 + 간단한 웹 UI
```

### 내가 검토해야 할 부분 (덕영)
- [ ] 현재 PLC에서 OPC-UA 지원하는지?
- [ ] 안 되면 Modbus나 다른 프로토콜은?
- [ ] Raspberry Pi로 충분한지, 산업용 게이트웨이가 필요한지?
- [ ] 실시간 데이터 수집 주기는 어느 정도가 적당한지?

## 8. 예상 Q&A

### Q1. 시간이 많이 드는 거 아니야?
> **A:** Phase 0~1은 기영이가 주도하고, 우리는 데이터 제공/검증 역할.
> 나(덕영)는 주말에 아키텍처 리뷰만, 준영이는 현업 하면서 필요시 피드백.

### Q2. 실패하면?
> **A:** Phase별 MVP 접근으로 작게 시작해.
> 1단계에서 효과 없으면 방향 수정하면 돼.
> 최소한 데이터 정리/문서화는 자산으로 남아.

### Q3. 외부 업체에 맡기면 안 돼?
> **A:** 가능하지만,
> 1) 우리 상황을 가장 잘 아는 건 우리
> 2) 벤더 종속되면 나중에 비용 폭탄
> 3) 유지보수 비용 지속 발생
> 4) 우리가 직접 하면 노하우 축적

### Q4. AI가 필요한 거 아니야?
> **A:** AI는 "보조 도구"야. 기본은 통계 + 룰 기반.
> 데이터 충분히 쌓인 후에 ML 도입 검토.
> "AI 쓴다"보다 "문제를 잘 정의하고 해결한다"가 핵심.
